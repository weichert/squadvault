## Summary

Nine `gate_*.sh` scripts use bare `sort` (or `sort -u`) on path strings
and other tokens but do not export `LC_ALL=C`. They pass inside
`prove_ci.sh`'s envelope (`prove_ci.sh:8` exports `LC_ALL=C`) but fail
or false-positive when invoked standalone from a default macOS shell
(`LANG=en_US.UTF-8`, no `LC_ALL`). This was the root cause of the
F3 apply script's mid-session "1 check FAILED" surprise on Steve's Mac
(commit `e358886` thread, when standalone gate verification was added
to the apply script for the first time).

This commit adds a single `export LC_ALL=C` line after `set -euo pipefail`
in each of the 9 gates. No other changes. 9 files, 9 insertions.

## Fix shape

The minimal, mechanical fix:

```bash
#!/usr/bin/env bash
set -euo pipefail
export LC_ALL=C   # NEW
```

Inserted as the third executable line after the shebang and `set -euo
pipefail`. This matches the precedent established by the (now-archived)
`gate_patch_wrapper_idempotence_allowlist_v1.sh` and the prior session's
hygiene bundle item 4 design intent for `gate_ci_proof_surface_registry_exactness_v1.sh`.

`LANG=C` is intentionally NOT exported. The active precedent in
`gate_prove_ci_structure_canonical_v1.sh` and the hygiene bundle's
intent are both `LC_ALL=C` only. `LC_ALL` overrides all other locale
variables; `LANG` would be redundant.

### Finding E (NEW) — Creative Surface Registry discoverability cluster

`scripts/gate_creative_surface_registry_discoverability_v1.sh` (lines
14-15) tests:

```bash
test "$c_marker" = "1"   # expects exactly one occurrence of: SV_CREATIVE_SURFACE_REGISTRY: v1
test "$c_bullet" = "1"   # expects exactly one occurrence of: docs/.../Creative_Surface_Registry_v1.0.md — Creative Surface Registry (canonical pointers) (v1)
```

Both tokens are absent from `docs/80_indices/ops/CI_Guardrails_Index_v1.0.md`.
Gate fails rc=1 silently with no echo output.

`scripts/gate_creative_surface_registry_usage_v1.sh:43` invokes the
above gate as a precondition; under `set -e` the parent gate inherits
the rc=1 and exits without a discernible error message.

This is structurally identical to **Finding C** from the F3 memo:

  - Finding C target: `gate_ci_proof_surface_registry_index_discoverability_v1.sh`
    expects `<!-- SV_CI_PROOF_SURFACE_REGISTRY: v1 -->` marker and a
    `check_ci_proof_surface_matches_registry_v1.sh` bullet, both absent.
  - Finding E target: `gate_creative_surface_registry_discoverability_v1.sh`
    expects `SV_CREATIVE_SURFACE_REGISTRY: v1` marker and a
    `Creative_Surface_Registry_v1.0.md` bullet, both absent.

Both are pre-existing at HEAD `c98bda6`. Both should be addressable in
a single follow-up commit ("discoverability bullets restoration"):
either ADD the missing marker+bullet pairs to the Index (preferred —
the gates serve a real purpose), or retire the gates if their value
is dubious.

I'd recommend the additive fix. The gates ARE valuable: they enforce
that the canonical registries are discoverable from the operational
index. The fix is two `<!-- marker -->` lines and two bullet entries
in the Index. Plus the surface fingerprint regeneration that follows
any Index change.
